# Business Ads Marketing

用于广告诊断、广告报表、全站推广加品/删品、品牌广告关键词推荐。候选域：`ads`。

## 意图路由

- 花费、曝光、点击、商机、ROI、趋势、排序：走广告报表。
- 为什么效果差、怎么优化、计划健不健康：走诊断；有 campaign ID 时优先计划诊断，否则账户诊断。
- 查计划、计划下商品/关键词/定向：走实体查询。
- 加品、删品、扩品、下架计划商品：先实体查询确认计划和关联关系，再写操作。
- 品牌广告关键词、打标、预定：走品牌广告关键词链路。

## 广告报表顺序

1. 解析时间，常见格式为 `yyyy-MM-dd HH:mm:ss`，不要和诊断日期格式混用。
2. 查 schema，确认数据源、filters、临时表、SQL 参数。
3. 先加载数据源，再执行查询。
4. 如果上一轮锁定了 `campaignId`、商品 ID 或关键词，本轮追问必须把 ID 放入 filters，避免查成全账户。
5. 多个互不依赖的报表 SQL、计划/商品/关键词效果查询，用 `workctl batch call` 并行执行；共享同一时间窗和 filters。
6. 输出金额时补币种，指标口径和时间窗必须写清。

## 诊断顺序

1. 判断账户诊断还是计划诊断。
2. 计划诊断需要 campaign ID；没有时先实体查询或向用户追问。
3. 账户诊断、计划诊断、报表补充数据互不依赖时，用 batch 并行读取；需要 campaign ID 的步骤先串行拿 ID。
4. 使用诊断工具的结论作为诊断来源，不要用报表数据自行伪造成“诊断结论”。
5. 输出问题、证据、影响、建议动作。

## 写操作顺序

1. 先实体查询，确认计划名、计划 ID、campaignType、subType、商品/adgroup 关系。
2. 去重、去 null，确保 ID 来自用户输入或上游工具返回。
3. 如果 schema 标记 `mutating` 或 `requires_yes`，展示计划、ID、subType、商品列表、操作类型和不可回滚风险。
4. 用户明确确认后执行；失败时读 `error.reason/error.next_action` 和业务 payload 错误。

## 品牌关键词顺序

1. 查店铺基础信息和商品行业。
2. 客户行为序列、近期高询盘词、高消耗词、高引流词属于只读 fan-out，优先 batch 并行查询。
3. 搜索关键词或次月释放资源。
4. 需要预定时先展示关键词、资源、预算/竞价含义和影响，再确认执行。

## 注意事项

- 不要把 token 当用户参数；鉴权由 workctl/runtime 处理。
- 同一会话保持全站推/标准推口径，不要未经用户切换混用。
- 报表、诊断、实体配置是不同能力；不要互相替代。
